# Avatar Model Sourcing Notes

Goal: use production-quality full-body basketball GLB assets suitable for the
`AvatarRenderStage` model path.

Current decision: do not ship generated, public low-poly, or downloaded
placeholder models as the avatar system. The runtime supports GLB rendering,
but checked-in generated GLBs are debug-only. The production path is
DCC-authored or provider-authored volumetric character rigs with explicit
`modelSource`, `modelQualityTier`, and `modelRigContract` metadata in the saved
avatar config.

## Runtime Import Contract

Provider/DCC avatar configs must include:

- `modelBaseRigAssetId`: stable reusable rig id for the selected base body.
- `modelUrl`: HTTPS `.glb` URL for provider-hosted models, or a bundled
  `assets/avatar_models/player_*.glb` path after production approval.
- `modelSource`: `externalGlb`, `avatarProvider`, or `productionDcc`.
- `modelQualityTier`: `modernGameRig`.
- `modelRigContract`: `hooprank_base_avatar_rig_v1`.
- `modelTopology`: `reusableBaseRig`.
- `modelCapabilities`: all values from
  `rig_contract.json.requiredRuntimeCapabilities`.
- Optional `modelLicense` and `modelAttribution` fields for auditability.
- `modelSpriteUrl`: transparent full-body PNG/WebP exported from the same rig
  for map markers.
- Camera metadata: `modelCameraOrbit`, `modelCameraTarget`,
  `modelFieldOfView`, and `modelScale`.
- A named idle clip in `modelAnimationName`.

Saved profile/map imagery should resolve to the GLB-derived `modelSpriteUrl`
or `modelPosterUrl` when present. The runtime suppresses generated/painter
avatar fallbacks and bundled starter sprites unless
the relevant local-debug flag is explicitly set. Static generated fallbacks
require `HOOPRANK_ALLOW_LOW_QUALITY_AVATAR_FALLBACKS=true`, procedural painter
fallbacks additionally require
`HOOPRANK_ENABLE_PROCEDURAL_AVATAR_PAINTER_FALLBACKS=true`, and bundled
starter sprites require `HOOPRANK_BUNDLED_CUSTOMIZED_AVATAR_SPRITES=true`.
Development base-rig GLBs and development starter sprites also require
`HOOPRANK_BUNDLED_DEVELOPMENT_AVATAR_MODELS=true` or
`HOOPRANK_BUNDLED_DEVELOPMENT_AVATAR_SPRITES=true` respectively, and the
matching source files must be intentionally added to `pubspec.yaml`.

The same suppression applies to individual accepted provider/DCC model configs
even when the global bundled-model flag is off. If a provider GLB is attached
but its sprite/poster export is missing, profile and map surfaces should expose
that missing production asset instead of downgrading to generated painter art.
New provider/DCC attachments created through the app helper must include a
production GLB, poster image, and transparent sprite image together.
SVG avatar imagery is not accepted for HoopRank avatar surfaces; it should be
re-exported as GLB plus PNG/WebP/JPG poster and sprite assets.

The app rejects `game_rig_*`, lit-relief, and prototype URLs unless
`HOOPRANK_DEBUG_PROTOTYPE_AVATAR_MODELS=true` is set. Bundled local production
slots also require `HOOPRANK_PRODUCTION_AVATAR_MODELS=true` until the catalog
passes the production art gate.

The current bundled six-base files named `player_<gender>_<build>_01.glb` are
generated by `tool/generate_avatar_base_rig_pack.py`. They are volumetric,
skinned, contract-shaped GLBs for runtime development and map/profile testing.
They are source/test fixtures until production approval; the release asset
manifest should not ship them while `catalog.status` remains
`awaiting_production_dcc_assets`.
They now include embedded procedural material and normal maps so the GLB path
exercises textured skin, hair, cloth, shoes, and basketball surfaces. They are
not final approved source art. Their current posters and map sprites are
mesh-projected from the same procedural parts for development parity only. The
entire set should be replaced by DCC-authored or provider-authored assets
before changing `catalog.json.status` to `production_dcc_assets`.

`rig_contract.json` is the scalable customization contract. `catalog.json`
declares exactly six `baseRigTargets`: skinny, medium, and big builds for male
and female bodies. Gender and build are base-body selectors, not morphs or
layers. Race/skin, hair, clothes, colors, height, and pose are runtime layers
applied to whichever base body is selected. `requiredSlots` define look
bindings against all six bodies. A compliant base rig must expose
HoopRank-named PBR material slots, hair/outfit visibility nodes, named idle
clips, and height morph targets. The Flutter runtime applies those layers
through the model-viewer scene graph rather than swapping low-resolution
generated images.

For provider/base-rig imports, outfit and pose changes preserve the attached
`modelUrl` and only update runtime controls and `modelAnimationName`. Gender
and build select the corresponding base body rig. This is the preferred
scalable path for avatar customization; look slots should remain layer
bindings, not separate character GLBs.

## Local DCC Import Flow

1. Export optimized GLB, poster, and transparent sprite files from the approved
   source scene.
   For Blender-authored packs, use the bundled exporter:

```bash
blender path/to/hooprank_avatar_pack.blend --background \
  --python tool/blender_export_avatar_model_pack.py -- \
  --output-dir assets/avatar_models/dcc_export
```

The exporter validates required material slots, hair/outfit/body visibility
nodes, shape-key morph targets, armature/skinned mesh presence, and required
idle animation clips before writing files.

2. Fill out `assets/avatar_models/import_manifest.example.json`, or use the
   manifest emitted by the Blender exporter.
3. Import into the Flutter bundle:

```bash
dart run tool/import_avatar_model_pack.dart --manifest path/to/import_manifest.json
dart run tool/validate_avatar_model_pack.dart --production-art
dart run tool/audit_avatar_model_pack.dart --markdown tmp/avatar_model_pack_qa.md --fail-on-blockers
```

4. After visual review, re-run the importer with `--production-approved` or set
   `catalog.json.status` to `production_dcc_assets`.

## Production Asset Bar

Required for the catalog to move from `awaiting_production_dcc_assets` to ready:

- Six reusable artist/provider-approved base body rigs matching
  `catalog.json.baseRigTargets`.
- Four look bindings matching `catalog.json.requiredSlots`.
- GLB exports matching the six approved base rigs.
- Six poster images matching `catalog.json.baseRigTargets`.
- Six transparent map sprites matching `catalog.json.baseRigTargets`.
- Original or commercially licensed character likenesses.
- No NBA/team/player/trademark dependency.
- Full-body athletic basketball player proportions.
- Rigged humanoid skeleton with skin weights and named idle clips.
- Transparent full-body map sprites exported from the same approved rig, not a
  separately drawn fallback.
- Named rig slots matching `assets/avatar_models/rig_contract.json`.
- PBR skin, hair, jersey, compression sleeve, shorts, sneaker, and ball
  materials.
- Tuned camera metadata saved in the avatar config.
- Idle animation clips named in `modelAnimationName`.
- GLB files passing `dart run tool/validate_avatar_model_pack.dart --production-art`.
- An audit report from
  `dart run tool/audit_avatar_model_pack.dart --markdown tmp/avatar_model_pack_qa.md --fail-on-blockers`
  with zero blockers.

Recommended paths:

- Commission or generate an original Blender/Maya/Unreal character pack.
- Export mobile-optimized binary GLB variants from the source DCC file.
- Store source art outside the Flutter bundle, but commit only optimized GLB and
  poster deliverables under `assets/avatar_models/`.
