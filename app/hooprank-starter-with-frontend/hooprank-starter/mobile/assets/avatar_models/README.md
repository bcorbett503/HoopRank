# HoopRank Avatar Model Assets

This directory is the production 3D avatar asset drop zone for profile setup.
The primary path is DCC-authored or provider-authored GLB characters: real
volumetric basketball players with rigged bodies, PBR skin/hair/cloth/shoe/ball
materials, tuned camera metadata, and idle animation clips.
Production GLBs must satisfy `rig_contract.json`, which defines the named
material slots, visibility nodes, and morph targets the app uses for scalable
runtime customization.

The app no longer treats pose-specific Python-generated `game_rig_*.glb` files
as acceptable production previews. Those files are debug artifacts only. The
current six `player_<gender>_<build>_01.glb` files are a stronger local
procedural base-rig pack generated by `tool/generate_avatar_base_rig_pack.py`:
they expose the runtime contract and embed procedural PBR-style material and
normal maps for skin, hair, cloth, shoes, and basketball surfaces. Each local
base rig exports a 35-joint humanoid skeleton with hands, face, feet, and toes,
but the pack is still not final artist-approved DCC/provider character art.

## Required Production Rig And Look Slots

`catalog.json` defines six reusable production base body rig targets: skinny,
medium, and big builds for male and female bodies. The four look slots are
bound to all six base bodies. Look slots select outfit visibility, material
colors, height morphs, camera metadata, and named idle animations; they should
not require separate character topology.

The four current look slots are layer bindings:

- `player_guard_01`: jersey, triple-threat idle
- `player_locked_in_01`: hoodie, crossed-arms idle
- `player_shooter_01`: warmups, jumper idle
- `player_dribble_01`: tee, dribble idle

The six production base bodies should include GLB, poster, and transparent
full-body map sprite files:

- `player_male_skinny_01.*`
- `player_male_medium_01.*`
- `player_male_big_01.*`
- `player_female_skinny_01.*`
- `player_female_medium_01.*`
- `player_female_big_01.*`

The current catalog status is `awaiting_production_dcc_assets` even though
local procedural six-base GLBs, posters, and map sprites are present. New
generated avatar configs do not attach model URLs by default because
`HOOPRANK_PRODUCTION_AVATAR_MODELS` defaults to `false`. This prevents the app
from showing non-approved procedural meshes in the avatar setup flow.

The local development posters and transparent map sprites are projected from
the same procedural base-rig mesh parts that build the GLBs. They exist to keep
profile, map, and validation paths wired to one six-base source of truth while
the final DCC/provider character art is pending.

To regenerate the local six-base development pack:

```bash
python3 tool/generate_avatar_base_rig_pack.py
dart run tool/validate_avatar_model_pack.dart --production-art
dart run tool/audit_avatar_model_pack.dart --markdown tmp/avatar_model_pack_qa.md --fail-on-blockers
```

The production-art model gate should fail while the catalog points at
`development_procedural_base_rig_v1`, while render review is not approved, and
while `catalog.json.status` remains `awaiting_production_dcc_assets`.
Artist/provider-approved source art should replace the procedural pack before
the catalog moves to `production_dcc_assets`.

The production GLB gate now checks measurable game-asset floors from
`rig_contract.json`, not only filenames or metadata. Each approved base body
must meet minimum vertex, triangle, joint, texture, normal-map, and skinned
mesh-node counts while staying under the mobile GLB, triangle, material, and
texture budgets.

Provider-hosted production avatars are allowed without that local bundle flag
when the saved avatar config explicitly includes:

- `modelBaseRigAssetId`: stable reusable rig id for the selected base body
- `modelSource`: `externalGlb` or `avatarProvider`
- `modelQualityTier`: `modernGameRig`
- `modelRigContract`: `hooprank_base_avatar_rig_v1`
- `modelTopology`: `reusableBaseRig`
- `modelCapabilities`: all capabilities listed in `rig_contract.json`
- `modelUrl`: HTTPS `.glb`
- `modelSpriteUrl`: transparent full-body PNG/WebP for map markers
- optional `modelLicense` / `modelAttribution`

Provider/base-rig avatars should keep the same `modelUrl` while users adjust
race/skin tone, hair, height, clothing, colors, and pose. Gender and build
select one of the six base body rigs. The Flutter runtime changes named
material colors, visibility nodes, height morph targets, and
`modelAnimationName`; it should not swap back to low-resolution generated
artwork once an approved provider rig is attached.

Once a generated avatar config has an accepted production/provider GLB attached,
the app treats missing `modelSpriteUrl` / `modelPosterUrl` as a production asset
gap instead of silently rendering generated fallback art. This keeps map and
profile surfaces honest: provider rigs must export the matching transparent map
sprite and poster from the same base rig. New provider/DCC attachments through
the app helper require both exports up front.

Avatar image surfaces intentionally reject SVG data URIs and remote `.svg`
files. Production avatars should be GLB for interactive preview and PNG/WebP/JPG
for exported poster or transparent map sprite imagery.

Enable production model previews only after the shipped GLBs pass the production
gates:

```bash
--dart-define=HOOPRANK_PRODUCTION_AVATAR_MODELS=true
```

Generated static avatar fallbacks, procedural painter fallbacks, and bundled
starter sprites are suppressed by default so low-quality stand-ins do not ship
as the avatar experience. Use these only for local debugging, and only after
intentionally adding the corresponding development asset files to
`pubspec.yaml`:

```bash
--dart-define=HOOPRANK_ALLOW_LOW_QUALITY_AVATAR_FALLBACKS=true
--dart-define=HOOPRANK_ENABLE_PROCEDURAL_AVATAR_PAINTER_FALLBACKS=true
--dart-define=HOOPRANK_BUNDLED_CUSTOMIZED_AVATAR_SPRITES=true
--dart-define=HOOPRANK_BUNDLED_DEVELOPMENT_AVATAR_MODELS=true
--dart-define=HOOPRANK_BUNDLED_DEVELOPMENT_AVATAR_SPRITES=true
```

The default shipping manifest must not include procedural development avatar
GLBs, starter sprites, or generated static avatar cutouts. Production-approved
imports can add the six base rig GLB/poster/sprite files back to the app bundle
after the production art gate passes.

## Importing A DCC Pack

Use `import_manifest.example.json` as the handoff schema from Blender/Maya,
Unreal, or an avatar provider export process. The importer copies files into
this directory and updates `catalog.json`.

For Blender-authored packs, run the exporter against a scene that already
contains the approved mesh, armature, material slots, visibility nodes, shape
keys, and animation clips:

```bash
blender path/to/hooprank_avatar_pack.blend --background \
  --python tool/blender_export_avatar_model_pack.py -- \
  --base-rig all \
  --output-dir assets/avatar_models/dcc_export
```

For a single `.blend` containing all six bodies, put each body in a collection
named exactly after its base rig id, for example `player_male_skinny_01` and
`player_female_big_01`. The exporter isolates one collection at a time when
writing each GLB, poster, and transparent map sprite. Blender may suffix
duplicate visibility node names such as `avatar_hair_fade.001`; during export,
the script temporarily normalizes selected contract node names so the generated
GLB still contains the exact names required by `rig_contract.json`. If a
`.blend` contains only one approved base body, pass that id with `--base-rig`.

Then import the generated manifest:

```bash
dart run tool/import_avatar_model_pack.dart --manifest assets/avatar_models/dcc_export/import_manifest.json
dart run tool/validate_avatar_model_pack.dart --production-art
dart run tool/audit_avatar_model_pack.dart --markdown tmp/avatar_model_pack_qa.md --fail-on-blockers
```

Only mark the catalog production-ready after visual review and validation:

```bash
dart run tool/import_avatar_model_pack.dart --manifest path/to/import_manifest.json --production-approved
```

Use this debug flag only for local GLB pipeline experiments:

```bash
--dart-define=HOOPRANK_DEBUG_PROTOTYPE_AVATAR_MODELS=true
```

## Acceptance Gates

Before enabling production avatar models:

```bash
dart run tool/validate_avatar_model_pack.dart --strict
dart run tool/validate_avatar_model_pack.dart --production-art
dart run tool/audit_avatar_model_pack.dart --markdown tmp/avatar_model_pack_qa.md --fail-on-blockers
dart run tool/validate_avatar_render_quality.dart --production-art
flutter analyze lib/utils/generated_avatar.dart lib/widgets/avatar_render_stage.dart lib/widgets/avatar_creator_sheet.dart lib/screens/profile_setup_screen.dart
flutter test test/generated_avatar_test.dart test/widgets/avatar_creator_sheet_test.dart test/widgets/avatar_image_test.dart
```

See [docs/avatar_production_asset_brief.md](../../docs/avatar_production_asset_brief.md)
for the full art brief and sourcing requirements.
